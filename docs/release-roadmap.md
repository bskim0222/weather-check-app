# 웨더체크 출시 로드맵

작성일: 2026-07-19

## 현재 상태

- 모바일 웹/PWA 형태의 앱 화면은 구현되어 있음.
- Render에 웹앱과 백엔드가 배포되어 있음.
- 기상청, 노르웨이 기상청, 핀란드 기상청 기준의 예보 비교 구조가 있음.
- 카카오 장소 검색과 카카오 지도 연동이 들어가 있음.
- 제보/문의/답변 UI는 있으나 저장 구조가 아직 출시용으로 안정적이지 않음.
- Android/iOS 앱스토어 출시를 고려한 Expo 설정은 있으나, 최종 앱 심사 단계까지는 아직 정리 필요.

## 출시까지 필요한 큰 작업

### 1. 데이터 신뢰성 고정

가장 먼저 해야 할 일.

- 요약탭과 비교탭이 같은 백엔드 응답을 사용하게 유지
- 요약카드 대표 상태, 기온, 문구가 3개 서비스 데이터와 어긋나지 않게 검증
- 검색 지역/시간 기준으로 모든 탭이 같은 컨텍스트를 공유
- 로딩 중에는 이전 지역 데이터를 새 지역 데이터처럼 보여주지 않기

### 2. 제보탭 서버/DB 안정화

출시 전 지인 테스트의 핵심.

- 현재 파일 기반 저장소를 정식 DB로 교체
- 문의글, 질문글, 답변글, 현장제보글을 분리 저장
- 글 작성/수정/삭제가 기기 간에 일관되게 반영
- 질문 답변 수가 실제 답변과 항상 맞도록 정리
- 샘플/가상 데이터는 테스트 배포에서 완전히 제거
- 신고/숨김 상태 관리

추천 DB 후보:

- Supabase PostgreSQL: MVP 테스트에 적합, 관리 화면 있음
- Render PostgreSQL: Render 안에서 묶어서 관리 가능

우선 추천은 Supabase. 이유는 관리자 화면에서 데이터를 직접 확인하기 쉽고, 나중에 로그인/권한 기능으로 확장하기 편함.

### 3. 지도탭 안정화

- 검색 지역과 현재 위치 핀 구분
- 지역별 제보 묶음 표시
- 지도 마커 클릭 시 바텀시트로 해당 묶음 글 표시
- 개인 위치가 드러나지 않도록 정확 좌표 대신 범위 묶음 사용
- 현재 위치 버튼 위치/디자인 정리

### 4. 앱 기본 기능 정리

- 첫 실행 시 위치 확인 후 자동으로 날씨 데이터 로딩
- 위치 권한 거부/실패 시 안내 화면 정리
- 새로고침은 현재 탭을 유지
- 로딩 화면/로딩 애니메이션 적용
- 오류 발생 시 “다시 시도” 흐름 제공

### 5. 앱 출시 준비

- 앱 이름, 로고, 스플래시, 아이콘 확정
- 개인정보 처리방침 작성
- 위치정보 이용 안내 문구 작성
- 사용자 신고/삭제 정책 작성
- Android 패키지명, iOS 번들 ID 최종 고정
- Android 내부 테스트 APK/AAB 빌드
- iOS TestFlight 준비

### 6. 지인 테스트

DB 안정화 후 진행.

- 5~10명에게 웹 링크 배포
- 검색 정확도 확인
- 날씨 데이터 신뢰도 확인
- 제보/문의/답변 동기화 확인
- 지도 제보 묶음 확인
- 작은 화면에서 레이아웃 깨짐 확인

## 당장 첫 작업

제보탭을 실제 테스트 가능한 상태로 만들기 위해 백엔드 저장 구조를 정리한다.

1. 현재 백엔드 저장 구조 확인
2. DB 도입 방식 결정
3. 데이터 모델 설계
4. 기존 API를 DB 기반으로 교체
5. 프론트 제보탭과 다시 연결
6. 웹 배포 후 다중 기기 테스트

## 우선순위

1. 제보탭 DB 안정화
2. 요약탭/비교탭 데이터 싱크 검증
3. 지도탭 제보 묶음 완성
4. 로딩/오류 UX 정리
5. 지인 테스트 배포
6. Android/iOS 앱 빌드 준비
