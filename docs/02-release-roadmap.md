# 앱 출시 로드맵 v0.1

이 문서는 웨더체크를 Android와 iOS 앱스토어에 올리기 위한 큰 순서입니다.

## 기술 선택

권장 방향은 React Native + Expo입니다.

쉽게 말하면, 한 번 앱을 만들고 Android용과 iPhone용으로 각각 포장할 수 있는 방식입니다.
Expo의 EAS Build를 사용하면 앱스토어 제출용 빌드를 만들 수 있습니다.

공식 참고:

- Expo EAS Build: https://docs.expo.dev/build/introduction/
- Expo 앱 제출: https://docs.expo.dev/submit/introduction/
- Apple Developer Program: https://developer.apple.com/programs/
- Google Play Console: https://play.google.com/console/about/

## 전체 진행 순서

### 1단계. 앱 설계 고정

완료에 가까운 단계입니다.

- 화면 구조 정리
- MVP 기능 정리
- 탭별 역할 정리
- 앱스토어 출시를 고려한 기술 방향 결정

### 2단계. Expo 프로젝트 생성

진행 완료.

현재 `mobile-app`이 Expo 기반 실제 앱 프로젝트입니다.
Android, iOS, web preview를 같은 코드에서 다룰 수 있습니다.

### 3단계. 목업을 앱 화면으로 옮기기

진행 중.

현재 아래 구조는 앱 코드로 옮겨졌습니다.

- AppHeader
- QuestionSearchBar
- DecisionCard
- EvidenceSection
- LocalReportForm
- MapViewMock
- ReportRequestList
- ForecastCompare
- BottomTabs

### 4단계. 앱 안에서 동작하게 만들기

진행 중.

처음에는 가짜 데이터로 시작합니다. 이 단계의 목표는 "누르면 화면이 바뀌고, 글을 쓰면 목록에 보이는 앱"입니다.

- 날씨 상태 변경
- 현장 제보 작성
- 제보 요청 선택
- 시간별/날짜별 비교 전환
- 검색 질문 입력
- 질문 기준 지도/제보/비교 화면 동기화
- 현재 위치와 검색 위치의 내부 구분
- 로딩/실패/목업 데이터 상태 표시

### 5단계. 실제 데이터 연결

화면이 안정된 뒤 실제 데이터를 붙입니다.

- 위치 권한
- 날씨 API
- 제보 저장 서버
- 지도 API
- 푸시 알림

이 단계부터 개인정보와 비용을 더 조심해서 봐야 합니다.

현재는 실제 API를 붙이기 전 준비로 아래 구조가 들어가 있습니다.

- API 환경 설정 파일
- 공통 API 클라이언트
- 날씨 예보 스냅샷 계약
- 현장 제보 스냅샷 계약
- 위치 참조 구조
- API 실패 시 목업 데이터로 대체 표시하는 UX

### 6단계. 테스트 빌드

사용자 휴대폰에 설치해서 확인합니다.

- Android 테스트 빌드
- iPhone 테스트 빌드
- 화면 깨짐 확인
- 위치 권한 문구 확인
- 느린 구간 확인

### 7단계. 스토어 제출 준비

앱스토어 심사에 필요한 자료를 준비합니다.

- 앱 이름
- 앱 아이콘
- 스크린샷
- 앱 설명
- 개인정보 처리방침
- 위치 정보 사용 설명
- 푸시 알림 사용 설명
- 테스트 계정

## 지금 바로 할 일

다음 작업은 실제 API를 바로 붙이기보다, 현재 샘플 데이터 앱을 더 "진짜 앱처럼" 다듬는 것입니다.

- 위치 권한 흐름 목업
- 로딩/실패/오프라인 상태
- 판정 결과 새로고침 흐름
- 제보 요청 작성/답변 UX 정리
- 앱스토어 스크린샷에 쓸 안정적인 화면 상태 만들기

"검색 질문을 넣으면 판정, 지도, 제보, 비교가 모두 같은 장소/시간 기준으로 움직이는 앱"까지 만들면 다음 큰 고비를 넘긴 것입니다.
